# Input Formats

This document describes the exact config format expected per vendor profile.

---

## cisco_ios — Cisco IOS / IOS-XE

**Command:** `show running-config`

Typical header: `Building configuration...` or `Current configuration : NNNN bytes`

```
version 17.6
hostname ROUTER-01
...
```

Detection signals: `Building configuration`, `version X.Y`, `line vty`, IOS interface names.

---

## cisco_asa — Cisco ASA

**Command:** `show running-config` or `more system:running-config`

Typical header: `: Saved` then `ASA Version X.Y`

```
: Saved
ASA Version 9.16(4)
hostname FW-01
...
```

Detection signals: `ASA Version`, `access-list … extended`, `object-group`, `nat (`.

---

## cisco_nxos — Cisco NX-OS

**Command:** `show running-config`

Typical header: `!Command: show running-config`

```
!Command: show running-config
version 10.2(4)
hostname NX-01
feature ssh
...
```

Detection signals: `!Command: show running-config`, `feature`, `vpc domain`.

---

## cisco_ftd — Cisco Firepower Threat Defense

Two accepted formats:

### ASA-shaped CLI (FDM-managed or FTD 6.x CLI)

Same format as `cisco_asa` with additional `# Firepower` or `NGFW Version` comment.

### FMC JSON export

JSON policy export from Firepower Management Center. Top-level keys include `type`, `name`, `accessRules`, etc.

```json
{
  "type": "AccessControlPolicy",
  "name": "MyPolicy",
  "accessRules": [...]
}
```

Detection: file extension `.json` or JSON content with `FirewallPolicy` type key.

**Note:** CIS controls requiring FMC visibility (IPS profiles, WildFire) are emitted as `manual`.

## arista_eos — Arista EOS

**Command:** `show running-config`

Typical header: `! device: HOSTNAME, EOS-X.Y.Z`

```
! device: EOS-SW-01, cEOSLab, EOS-4.30.1F
hostname EOS-SW-01
...
```

Detection signals: `! device: … EOS`, `! image version:`, `daemon TerminAttr`.

---

## hpe_aruba — HPE Aruba / ProCurve

**Source:** Web UI download or `show running-config`

Typical header: `; JNNNNX Configuration Editor`

```
; J9726A Configuration Editor
Running configuration:
hostname "ARUBA-SW"
...
```

Detection signals: `; JNNNNX`, `Running configuration:`, ProCurve STP/VLAN syntax.

---

## fortios — Fortinet FortiOS

**Source:** Full config backup via GUI or `execute backup config flash`

Typical header: `#conf_file_ver=…` or `#config-version=FGVM…`

```
#conf_file_ver=2200000002
config system global
    set hostname "FW-01"
    ...
end
```

Detection signals: `config system global`, `set … / next / end`, `#conf_file_ver`.

---

## junos — Juniper JunOS

**Primary format:** `show configuration | display set` (flat set-statement output)

```
set version 23.2R1
set system host-name ROUTER-01
set system services ssh
...
```

**Alternate (Phase 1 stretch goal):** curly-brace hierarchical format from `show configuration`

Detection signals: `set system host-name`, `set version`, `## Last commit:`.

---

## palo_alto — Palo Alto Networks PAN-OS

**Source:** Full XML config export (`scp` or Panorama push)

```xml
<?xml version="1.0"?>
<config version="11.0.0" urldb="paloaltonetworks">
  <devices>
    <entry name="localhost.localdomain">
      <deviceconfig>
        <system><hostname>PA-FW-01</hostname></system>
        ...
```

Detection signals: `<config version=`, `<vsys>`, `<rulebase>`.

---

## checkpoint — Check Point

**Primary format:** Gaia CLISH export or `clish -c "show configuration"`

```
set hostname CP-GW-01
set ssh enable
set ntp server primary 10.0.0.1
...
```

**Alternate:** SmartConsole object dump (`:name (` syntax)

Detection signals: `set ssh`, `set hostname`, `add access-rule`, `:name (`.

**Note:** Most policy-layer controls require SmartConsole access and are emitted as `manual`.

---

## sophos_xg — Sophos XG / SFOS

**Source:** Full XML backup exported from web console (Backup & Firmware > Backup)

```xml
<?xml version="1.0" encoding="UTF-8"?>
<XGFirewallConf APIVersion="2101.1">
  <Administration>
    <HostName>XG-FW-01</HostName>
    ...
```

Detection signals: `<XGFirewallConf>`, `<FirewallRule>`, `<Administration>`.

---

## sonicwall — SonicWall SonicOS

**Accepted formats:**
- Text config export (SonicOS 7.x Enhanced CLI)
- XML `.exp` backup

```
firmware-version SonicOS Enhanced 7.0.1
management https
management ssh
...
```

Detection signals: `firmware-version SonicOS`, `access-rule from`, `address-object`.

---

## nokia_sros — Nokia SR OS

**Source:** `admin save` output (classic CLI model)

Typical header: `# TiMOS-B-X.Y.RN …`

```
# TiMOS-B-22.10.R2 both/x86_64 Nokia 7750 SR
configure {
    system {
        name "SR-RTR-01"
        ...
```

Detection signals: `# TiMOS`, `configure {`, `# Generated by SR OS`.

---

## nokia_srl — Nokia SR Linux

**Source:** JSON config export via gNMI or `info flat | display json`

```json
{
  "srl_nokia": {
    "system": {
      "name": { "host-name": "SRL-NODE-01" },
      ...
```

Detection signals: `"srl_nokia"`, `"network-instance"`, `"interface"`.

---

## Automation vs. Manual

Each vendor YAML file contains automated probes (regex or Python hook).  
The `cis_catalog.yaml` lists controls that **cannot** be automated from static config and are always emitted as `status: manual`.  
The `% compliance_score` in the summary covers only automated checks.
